# Documentation guidelines

Docs here are read by busy humans. Keep them short.

- One topic per file. Need a second topic? Write a second file.
- Aim for one screen. A doc that scrolls for pages has failed.
- Lead with what the reader does. Cut history and rationale unless they change an action.
- Prefer a table, a command, or a short list over prose.
- Show a working example instead of describing one.
- Don't restate what's elsewhere — link to the source of truth.

If you can't say it briefly, link to the code and explain only the part that isn't obvious.
